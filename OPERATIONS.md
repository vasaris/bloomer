# OPERATIONS — эксплуатация Blumer Bot

Шпаргалка «как сопровождать бота через год, не вспоминая с нуля». Стек: **aiogram 3.x · SQLite (aiosqlite) · APScheduler · Railway · Claude API (опц.)**. Процесс — `worker` (long polling, без публичного порта), `Procfile: worker: python -m bot.main`.

## Команды в боте (быстро)

| Команда | Что делает |
|---|---|
| `/ping` | тест-пуш (минуя тихие часы) — проверка живости |
| `/today` · `/weekly` | бриф дня · недельный обзор по запросу |
| `/stats` | уровень, XP, стрики, заморозки, ачивки |
| `/backup` | бэкап БД сейчас + файл в чат |
| `/export` | выгрузка журнала (zip: CSV по таблицам + lossless JSON) |
| `/settings` | текущее время всех пушей и тихих часов |
| `/settime код ЧЧ:ММ\|off` · `/setquiet` · `/setweekly` | правки расписания (применяются сразу, без рестарта) |
| `/arrived [ГГГГ-ММ-ДД]` | дата приезда Блумера — включает весь распорядок |

## Логи

- Railway → сервис → **Deployments → View Logs** (или **Observability → Logs**). CLI: `railway logs`.
- При старте в логах: `Зарегистрирован пуш '<code>' на HH:MM`, `Зарегистрирован недельный обзор: …`, `Зарегистрирован бэкап БД на HH:MM …`, `Планировщик запущен`, `BLUMER BOT v<версия> — polling...`.
- Полезные маркеры в работе: `Пуш '<code>' пропущен — тихие часы`, `… пропущен — сегодня не нужен`, `БД забэкаплена → …`, `Пуш '<code>' перепланирован на HH:MM`.
- Уровень — INFO. Трейсбек при падении ищи в начале последнего деплоя.

## Сменить модель Claude

- Переменная окружения `CLAUDE_MODEL` (Railway → **Variables**). Дефолт — `claude-sonnet-4-6`.
- Изменил → Railway передеплоит автоматически. Проверка: задай вопрос через `/ask`.
- Нет `ANTHROPIC_API_KEY` → `/ask` вежливо подскажет, остальные команды работают как обычно.

## Бэкапы

- **Авто:** ежедневно в `BACKUP_TIME` (дефолт 03:30), хранится `BACKUP_KEEP` (дефолт 7) последних копий в `backups/` рядом с БД. При `BACKUP_TO_TELEGRAM=on` (дефолт) — тихая офсайт-копия `.db` приходит в чат.
- **Вручную:** `/backup`.
- **Отключить авто:** `BACKUP_TIME=off`.
- Копия делается через SQLite online-backup (целостный снимок даже под записью), не сырым копированием файла.

## Восстановить БД из бэкапа

1. **Останови запись** — поставь сервис на паузу / останови деплой в Railway (чтобы не подменять файл под активной записью).
2. Возьми нужный бэкап: файл из `backups/` на Volume **или** `.db`, который бот присылал в Telegram.
3. Замени файл по пути `DB_PATH` этим бэкапом (имя файла = то, что в `DB_PATH`).
   - Railway shell: `cp backups/blumer-YYYYMMDD-HHMMSS.db "$DB_PATH"`.
   - Из Telegram-копии: скачай `.db` и положи на Volume на место `DB_PATH`.
4. Запусти сервис. Схема докатится сама — `init_db` идемпотентен (добавляет недостающие колонки/таблицы).

## Если бот молчит

По убыванию вероятности:

1. **Блумер ещё не «приехал» (by design).** До `/arrived` операционные пуши (прогулки, кормёжка, нюхо, груминг, здоровье, астма, итоги, обзор) **намеренно молчат** — говорит только утренний бриф с обратным отсчётом. Проверь `/adaptation`; лечится `/arrived` (в день приезда) или `/arrived ГГГГ-ММ-ДД`.
2. **Тихие часы.** В окне `QUIET_HOURS` пуши пропускаются (в логах «пропущен — тихие часы»). Проверь `/settings`. `/ping` тихие часы игнорирует: если `/ping` приходит, а обычные пуши нет — дело в тихих часах или расписании.
3. **Worker не запущен / упал.** Railway → статус деплоя; в логах ищи трейсбек. Частая причина — не задан `BOT_TOKEN`.
4. **Пуш «не нужен сегодня».** `health_check` / `groom_check` / `asthma_check` шлются только когда есть что сказать (по графику / в окне 21 дня адаптации). Это норма, не баг.
5. **Время сдвинуто/выключено.** `/settings` — проверь, не стоит ли «выкл» или непривычное время. Правки из бота имеют приоритет над `.env`.
6. **БД обнулилась.** Пропали стрики/логи/настройки после редеплоя → `DB_PATH` не на Volume (см. ниже).
7. **Доступ.** Бот отвечает только whitelisted `chat_id` (`OWNER_CHAT_IDS`). Сообщение «Доступ ограничен» — id не в списке.

## Персистентность (Volume) — критично

Вся память бота (журнал, стрики, XP, ачивки, дата приезда, **настройки времени пушей**, бэкапы) — в одном SQLite-файле по `DB_PATH`. Контейнер на Railway эфемерный, поэтому:

- Подключи **Volume** и укажи `DB_PATH` внутрь него (например `/data/blumer.db`). Тогда и БД, и `backups/` рядом переживут редеплой.
- Офсайт-копии в Telegram — второй контур на случай потери Volume.

Проверка: после редеплоя `/stats` и `/settings` должны показывать прежние данные. Если обнулилось — Volume не подключён или `DB_PATH` мимо него.

## Переменные окружения

Полный список с дефолтами — в `.env.example`. Минимум для старта: `BOT_TOKEN`, `OWNER_CHAT_IDS`, `TIMEZONE`. Группы: расписание `PUSH_*`, тихие часы `QUIET_HOURS_*`, бэкапы `BACKUP_*`, погода `LATITUDE`/`LONGITUDE`, Claude `ANTHROPIC_API_KEY`/`CLAUDE_MODEL`. Время пушей и тихих часов лучше править из бота (`/settings`) — это переживает рестарт и не требует редеплоя.

## Деплой / откат

- Деплой = `git push` в `main` → Railway авто-редеплой.
- Откат кода: Railway → **Deployments** → выбрать прошлый успешный → **Redeploy/Rollback**; либо `git revert` + push.
- Миграции идемпотентны (применяются при старте), отдельных шагов вручную не требуют.
